编译通过镇一下哈哈

Bug1

  1. error LNK2001: 无法解析的外部符号 "__declspec(dllimport) const XXX::vftable"
  2. 无法定义 dllimport 实体

这两个都是由于宏定义引起的。我的工程中是这样定义的

直接说最终的结果，在几经百度后知道__declspec(dllimport)与__declspec(dllexport)是用来在引用hpp文件的时候就可以使用对应cpp文件功能的。

百度解释：

__declspec(dllexport)：导出符号，定义需要导出函数的dll中给导出函数的函数声明前面加上导出符号，表示该方法可以导出给其他DLL或者exe使用
__declspec(dllimport):导入符号，在使用该函数的DLL或者exe中需要在该函数的函数声明前面加上该符号，表示该函数方法是从其他库导入的

按照这个思路，定义完成后就该使用了。
使用：

这样是直接抛出异常了，原因是在hpp中定义了宏之后并没有在cpp文件中定义

在最开始添加#define EXPORTSS 就可以了。

又有新Bug

无法解析的外部符号，对应的错误实在是太多了。
不外乎就是

  1. 查C/C++常规中的附加包含目录是不是添加到了include。
  2. 链接器常规中附加包含目录是不是引用了库的路径
  3. 链接器输入中是不是写完了所有引用的lib文件
  4. 所有头文件中的引用是不是成功
  5. 提前定义无法定义 dllimport 实体的类型
    问题

    解决

    这次真的是不知道问题在哪了，所有改查的都查了，在查文件的时候发现，之前有一个在GeneratedFiles文件下面的 ..moc...文件报错的，当时把这个文件删了，就没报错了。安全起见我把当时备份在压缩包的文件重新加了进去。
    编译…..通过了

谁知道。。。。竟然是这样的。。。这随手备份的好习惯。。

可是为什么么呢？

百度表示：moc 全称是 Meta-Object Compiler，也就是“元对象编译器”。Qt 程序在交由标准编译器编译之前，先要使用 moc 分析 C++ 源文件。如果它发现在一个头文件中包含了宏 Q_OBJECT，则会生成另外一个 C++ 源文件。这个源文件中包含了 Q_OBJECT 宏的实现代码。这个新的文件名字将会是原文件名前面加上 moc_ 构成。这个新的文件同样将进入编译系统，最终被链接到二进制代码中去。
